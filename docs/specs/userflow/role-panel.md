# Role Panel 유저플로우

> 역할 패널(role-panel) 도메인의 유저플로우. 관리자가 웹 대시보드(`/settings/guild/[guildId]/role-panel/`)에서 패널을 생성·설정·게시하고, 일반 사용자가 Discord 채널에 게시된 패널 버튼을 클릭해 역할을 부여받거나 토글하는 흐름을 다룬다. 봇 인터랙션 핸들러(`apps/bot/src/event/role-panel/`)와 API 비즈니스 로직(`apps/api/src/role-panel/`), 웹 설정 페이지(`apps/web/app/settings/guild/[guildId]/role-panel/`)가 연동된다.
> 🔒 마커: 역할 부여 불가 처리(봇 역할 위계·managed·ADMINISTRATOR 역할 차단), Discord 권한 부족 처리 줄.

> 웹 표면 플로우(UF-ROLE-PANEL-001~005)는 `_index.md`의 "공통 전제 (웹 대시보드 진입)" 흐름을 전제한다.
> 봇 표면 플로우(UF-ROLE-PANEL-006~008)는 `_index.md`의 "공통 전제 (봇 슬래시 커맨드 / 버튼 상호작용)" 흐름을 전제한다.

## UF-ROLE-PANEL-001: 역할 패널 목록 조회 (웹)

### 입력
- 관리자가 역할 패널 설정 페이지에 접근

### 처리
1. (공통 전제: 웹 대시보드 진입 + 길드 운영진 또는 슈퍼관리자 권한 검증 통과)
2. API로 현재 길드의 역할 패널 목록(패널 기본정보 + 버튼 목록) 조회
3. 조회 결과를 탭 형태로 렌더링 (패널별 이름, 게시 상태, 대상 채널)
4. 비운영 길드 슈퍼관리자인 경우 read-only 상태임을 화면에 표기

### 출력
- 패널 목록 표시 (등록된 패널 없으면 빈 상태 표시)
- 비운영 길드 접근 시 read-only 안내

### 엣지케이스
- 비운영 길드 슈퍼관리자 접근 → 조회는 정상 진행, read-only 상태 표기 (mutation 시 API 403 예정)
- 봇이 해당 길드에 미참여 상태 → 목록 조회는 가능하나 게시 불가 안내 표기
- 네트워크 오류 또는 API 응답 실패 → 오류 안내 표시, 재시도 유도

## UF-ROLE-PANEL-002: 새 패널 생성 및 설정 저장 (웹)

### 입력
- 관리자가 새 패널 버튼을 클릭하고, 패널 이름·대상 채널·Embed 설정(제목/설명/색상)·버튼 목록(라벨/역할/모드)을 입력한 후 저장

### 처리
1. (공통 전제 통과)
2. 역할 선택기 렌더링 시 길드 역할 목록에서 🔒 부여 불가 역할(봇 역할보다 높은 역할 / @everyone / managed·integration 역할)은 비활성·제외 표기, 🔒 ADMINISTRATOR 권한 보유 역할은 비활성·차단 사유 표기
3. 저장 요청 전 클라이언트 유효성 검사: 버튼 0개 차단, 버튼 25개(ActionRow 5 x 5) 초과 차단
4. API가 역할 부여 가능 여부(봇 역할 위계·managed 여부) + ADMINISTRATOR 비트를 서버 측 재검증
5. 검증 통과 시 패널 정보(published = false)와 버튼 목록을 DB에 저장
6. 캐시 무효화

### 출력
- 저장 성공: 성공 안내, 패널이 목록에 미게시 상태로 추가됨
- 저장 실패: 오류 안내

### 엣지케이스
- 🔒 부여 불가 역할 선택 시도 → UI 비활성으로 선택 불가, API 검증에서도 차단
- 🔒 ADMINISTRATOR 보유 역할 선택 시도 → UI 비활성 + 차단 사유 표기, API 403 응답
- 버튼 0개 저장 시도 → 클라이언트에서 저장 전 차단
- 버튼 25개 초과 → 클라이언트에서 저장 전 차단 (API에서도 400 반환)
- 비운영 길드 슈퍼관리자 저장 시도 → API 403

## UF-ROLE-PANEL-003: 패널 게시 — 신규 게시 (웹)

### 입력
- 저장된(미게시) 패널에서 게시 버튼 클릭

### 처리
1. (공통 전제 통과)
2. 대상 채널 미선택 상태이면 저장 단계에서 차단 (게시 전 저장 필수)
3. API가 publish 엔드포인트를 호출하여 Bot-API-Client 경유 봇에 게시 요청
4. 봇이 대상 채널에 Embed + 버튼 메시지를 신규 전송
5. Discord가 반환한 messageId를 DB에 저장, published = true로 갱신
6. 캐시 무효화

### 출력
- 게시 성공: 성공 안내 토스트, 패널 상태가 게시됨으로 갱신
- 게시 실패: 오류 안내 (원인별 안내 문구 분기)

### 엣지케이스
- 대상 채널이 삭제되어 존재하지 않음 → 봇 오류 응답 전달 → 채널 재선택 유도 안내
- 🔒 봇이 해당 채널에 메시지 전송 권한 부족(Send Messages 없음) → 봇 503 응답 → 권한 부족 안내
- 봇 자체 오프라인 또는 Bot-API-Client 통신 실패 → 오류 안내
- 대상 채널 미선택 상태로 게시 시도 → 저장 단계에서 차단되므로 게시 단계 도달 불가

## UF-ROLE-PANEL-004: 기존 패널 수정 및 재동기화 (웹)

### 입력
- 기존 패널을 선택하고 설정(이름·채널·Embed·버튼 목록)을 변경한 후 저장

### 처리
1. (공통 전제 통과)
2. 변경 내용 클라이언트 유효성 검사 (버튼 개수, 역할 매핑 가드)
3. API가 역할 부여 가능 여부 + ADMINISTRATOR 비트 재검증
4. 검증 통과 시 DB에 패널 정보 업데이트 (버튼 목록 전체 교체)
5. published = true 상태이면 봇에 메시지 동기화 요청
6. 채널이 변경된 경우: 기존 채널의 Discord 메시지 삭제 → 새 채널에 신규 전송 → messageId 갱신
7. 채널이 동일한 경우: 기존 messageId를 사용하여 Discord 메시지 edit (삭제 후 재전송 아님)
8. 캐시 무효화

### 출력
- 저장·동기화 성공: 성공 안내, Discord 채널 메시지가 변경 내용으로 갱신됨
- 저장 실패 또는 동기화 실패: 오류 안내

### 엣지케이스
- 기존 Discord 메시지가 수동 삭제됨 → edit 실패 → 해당 채널에 신규 전송으로 폴백 후 messageId 갱신
- 채널 변경 시 기존 채널 메시지 삭제 실패(이미 삭제됨) → 내부 로그 기록 후 새 채널 신규 전송 계속 진행
- 🔒 수정 중 역할 검증 실패(부여 불가 역할 또는 ADMINISTRATOR 역할 추가 시도) → API 거부, 오류 안내
- 비운영 길드 슈퍼관리자 저장 시도 → API 403

## UF-ROLE-PANEL-005: 패널 삭제 (웹)

### 입력
- 패널 삭제 버튼 클릭 후 확인 다이얼로그에서 삭제 확인

### 처리
1. (공통 전제 통과)
2. messageId가 존재하면 해당 채널 Discord 메시지 삭제 시도
3. 메시지 삭제 성공·실패와 무관하게 DB 삭제 계속 진행 (실패 시 내부 로그 기록)
4. DB에서 RolePanelButton 삭제 → RolePanelConfig 삭제 (순서 준수)
5. 캐시 무효화

### 출력
- 삭제 완료: 완료 안내, 패널 목록에서 제거됨
- (Discord 메시지 삭제 실패는 사용자에게 별도 오류 노출 없이 내부 로그로만 처리)

### 엣지케이스
- Discord 메시지가 이미 수동 삭제된 상태 → 삭제 시도 무시, DB 삭제 계속 진행
- 비운영 길드 슈퍼관리자 삭제 시도 → API 403

## UF-ROLE-PANEL-006: GRANT 모드 버튼 클릭 — 역할 부여 (봇)

### 입력
- 사용자가 게시된 패널의 GRANT 모드 버튼을 클릭

### 처리
1. (봇 공통 전제: 버튼 인터랙션 수신, 길드 컨텍스트 확인, customId 접두사로 핸들러 분기)
2. customId(`role_panel:{panelId}:{buttonId}`) 파싱으로 panelId, buttonId 추출
3. Redis 캐시 조회(캐시 미스 시 DB 조회 + 캐시 갱신)로 버튼 설정(roleId, mode) 조회
4. 사용자가 해당 역할을 이미 보유 중이면 Discord API 호출 없이 Ephemeral 안내 후 종료 (멱등 처리)
5. 미보유 시 Discord API로 역할 부여 (GuildMember.roles.add)
6. Ephemeral 성공 응답 전송
7. (선택) 역할 부여 이벤트 내부 로그 기록

### 출력
- 역할 미보유 → 부여 성공: Ephemeral 부여 성공 안내 (본인에게만 보임)
- 역할 이미 보유(멱등): Ephemeral 이미 보유 안내 (본인에게만 보임)
- 오류: Ephemeral 오류 안내 (본인에게만 보임)

### 엣지케이스
- 🔒 봇에 Manage Roles 권한 없거나 봇 역할이 대상 역할보다 낮음 → Discord API 실패 → Ephemeral 권한 부족 안내
- 해당 역할이 삭제된 상태(미싱 역할) → Ephemeral 오류 응답 + 내부 로그
- customId 파싱 실패(잘못된 형식) → Ephemeral 잘못된 요청 안내
- 버튼 설정이 DB에 없음(패널 삭제됨) → Ephemeral 오류 안내
- Discord 응답 3초 초과 위험 시 deferReply + followUp 패턴으로 처리
- DM 컨텍스트 인터랙션 → 무시 또는 길드 전용 안내

## UF-ROLE-PANEL-007: TOGGLE 모드 버튼 클릭 — 역할 토글 (봇)

### 입력
- 사용자가 게시된 패널의 TOGGLE 모드 버튼을 클릭

### 처리
1. (봇 공통 전제 동일)
2. customId(`role_panel:{panelId}:{buttonId}`) 파싱으로 panelId, buttonId 추출
3. Redis 캐시(캐시 미스 시 DB 조회) 버튼 설정(roleId, mode) 조회
4. 동시성 보호: Redis 락(키: `role_panel:lock:{guildId}:{userId}:{buttonId}`, TTL 3초) 획득 시도 → 획득 실패 시 이미 처리 중 Ephemeral 안내 후 종료
5. 사용자가 역할을 보유 중이면 역할 회수 (GuildMember.roles.remove) → Ephemeral 회수 안내
6. 미보유이면 역할 부여 (GuildMember.roles.add) → Ephemeral 부여 안내
7. Redis 락 해제

### 출력
- 역할 보유 → 회수 성공: Ephemeral 회수 안내 (본인에게만 보임)
- 역할 미보유 → 부여 성공: Ephemeral 부여 안내 (본인에게만 보임)
- 오류: Ephemeral 오류 안내 (본인에게만 보임)

### 엣지케이스
- 🔒 봇 Manage Roles 권한 부족 또는 봇 역할이 대상 역할보다 낮음 → Discord API 실패 → Ephemeral 권한 부족 안내
- 빠른 연속 클릭(레이스 컨디션): Redis 락으로 동시 처리 차단 → 두 번째 요청자에게 처리 중 Ephemeral 안내
- 역할이 삭제된 상태(미싱 역할) → Ephemeral 오류 응답 + 내부 로그
- Discord 3초 응답 제한 위험 시 deferReply + followUp 패턴으로 처리
- DM 컨텍스트 인터랙션 → 무시 또는 길드 전용 안내

## UF-ROLE-PANEL-008: 인증 게이트 — 규칙 동의 버튼 클릭 (봇, IA §4)

### 입력
- 미인증(@everyone) 상태 사용자가 `규칙-rules` 채널에 게시된 GRANT 모드 동의 버튼을 클릭

### 처리
1. (봇 공통 전제 동일)
2. UF-ROLE-PANEL-006 GRANT 모드 흐름과 동일하게 처리 (정회원 역할이 roleId로 매핑된 경우)
3. 미인증 사용자(@everyone)는 IA §3 권한 매트릭스에 따라 `규칙-rules` 채널에서 버튼 상호작용 가능
4. 정회원 역할 미보유 확인 → Discord API로 정회원 역할 부여 (GuildMember.roles.add)
5. Ephemeral 성공 응답 전송 (본인에게만 보임)
6. 정회원 역할 부여로 커뮤니티·음성·게임·지원 카테고리 접근 권한 자동 개방 (Discord 채널 권한 상속 구조에 의함)

### 출력
- 정회원 역할 미보유 → 부여 성공: Ephemeral 정회원 역할 부여 안내 (본인에게만 보임)
- 이미 정회원(멱등): Ephemeral 이미 보유 안내
- 오류: Ephemeral 오류 안내

### 엣지케이스
- 이미 정회원 역할 보유 → GRANT 멱등 처리, Ephemeral 이미 보유 안내 (UF-ROLE-PANEL-006 동일)
- 🔒 봇 역할이 정회원 역할보다 낮게 배치됨 → 역할 부여 실패, Ephemeral 권한 오류 안내 (관리자가 봇 역할 위계 조정 필요)
- `규칙-rules` 채널 패널이 삭제된 후 버튼 클릭 → 버튼 설정 없음 Ephemeral 안내

## 후속 확장 (MVP 외)

- **EXCLUSIVE(택1) 모드**: 한 패널 내 버튼 중 하나만 역할 보유하도록 강제. 다른 버튼 클릭 시 기존 역할 자동 회수 후 신규 역할 부여. 역할 간 상호 배타 처리 복잡도 및 레이스 컨디션 대응으로 다음 Phase에서 다룬다. MVP에서는 제외.
- **채널별 Discord 역할 부여 로그 (감사 로그)**: MVP는 내부 시스템 로그만 기록. 추후 지정 채널에 역할 부여·회수 이벤트를 Discord Embed로 발송하는 방식으로 확장.
- **역할 조건부 제한**: 특정 역할을 보유한 사용자만 버튼 클릭을 허용하는 사전 역할 요건 기능. 후속 Phase에서 다룬다.
